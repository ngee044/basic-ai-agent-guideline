# ~/.claude/CLAUDE.md (global)

모든 Claude Code 세션에 자동 로딩되는 사용자 전역 지침. 프로젝트 별 세부 규칙은 해등 프로젝트 루트의 'CLAUDE.md'에 둔다

## 응답
- 한국어로 답변
- 불필요한 서론/요약 생략
- 항상 격식을 갖추어서 답변해주세요. (존댓말 사용)
- 임의로 커밋, PR 생성은 하지 않습니다. 사용자가 커밋 및 PR 생성을 직접 합니다. 단, 사용자 요청에 따른 행동은 유효합니다.
- 비교/분석에 대한 답변 시 표를 사용하여 가독성을 높입니다.
- 상세한 답변 요구를 하지 않으면 간략하게 설명하여, 현재 상황/상태에 대해 빠른 파악을 위해 간략하게 설명합니다.

# 기본 분석 규칙
1. 모든 프로젝트 분석은 ISO 기반으로 문서 작성을 통해 분석합니다.
2. ISO 문서로 적합성 및 추적성을 고려하여 작성합니다. 특히 추적성의 우선순위는 0 티어 입니다.
3. 실제 문서와 프로젝트 코드 레벨을 검증합니다.
4. 문서 저장은 프로젝트 하위 docs/ 폴더 내부에 저장합니다.
5. .gitignore에 docs/ 폴더 경로를 추가하여 문서 관련 내용이 커밋 되지 않도록 관리합니다.

# 코드 작성
- 각 프로젝트 docs/ 하위 폴더에 코드 규약 문서가 있는지 확인합니다. 그리고 이를 따릅니다. 규약이 없는 경우 코드 규약 문서 작성에 대해서 제안해주세요.
- C/C++의 경우 명확한 규약이 존재합니다. 그러나 Golang, python, javascript 등 가장 일반적으로 사용하는 규약을 사용합니다. 언어 특성을 고려하여 규약을 정해서 사용해주세요.
- 주석은 WHY가 비자명할 때만 해주세요.
- 추측 금지: 모르면 검색 확인 후 답변 합니다.
- 검색 및 확인 코드 작성 시에 토큰양에 제한을 두지 않습니다.
- 질문이 전달력이 부족하거나 부정확한 지시가 있다고 판단되는 경우 상세 질문을 할 수 있도록 유도해주세요 (되물어보기)
- 컴파일이 가능한 언어(프로젝트)의 경우 반드시 컴파일하여 컴파일 에러가 없어야 합니다.
- 각 프로젝트의 코드 규약(Convention)과 외부 라이브러리 문서가 존재하는지 확인하고 이를 참고하여 코드를 작성합니다.
-- 외부 라이브러리 문서가 없는 경우 검색을 통하여 API 문서를 확인해주세요. 자체 라이브러리의 경우 Samples 폴더가 존재할겁니다. 해당 폴더에 참고할만한 코드가 있으니 이를 참고 합니다.
- '{target_project}/docs'의 내부에 대체로 ISO 문서 및 기타 프로젝트 관련  내용이 존재합니다. 이를 통해 코드 작성 및 검증을 합니다.
- feature, bug fix 등 수정 시 ISO 문서 갱신이 필요한 경우 진행합니다.

# 아키텍처(MSA)
- 새 프로젝트/시스템은 modular-monolith-first로 시작합니다. 처음부터 마이크로서비스로 쪼개지 말고, 경계(bounded context)·데이터 소유권·계약(contract)을 명확히 설계하여 필요할 때 무중단으로 서비스 추출이 가능하도록 합니다.
- 서비스 분리(추출)는 독립 배포 주기, 팀 소유권 분리, 부하/확장 특성 차이, 장애 격리, 기술 스택 분리 등 정당한 사유가 다수 성립할 때만 진행합니다("코드가 커져서"는 사유가 아닙니다).
- 분산 시스템의 관심사(계약·멱등성·회복탄력성·관측성·독립 배포)는 MSA 가이드라인을 따릅니다. 분산 모놀리스(동기 강결합 + 공유 DB)는 지양합니다.
